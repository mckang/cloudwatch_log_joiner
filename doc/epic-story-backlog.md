# CloudWatch Analysis Epic & Story Backlog

## 문서 목적
- `prd.md`를 기반으로 개발/운영 팀이 바로 실행 가능한 제품 백로그를 정의한다.
- Epic 단위의 목표와 User Story 단위의 완료 기준(수용 기준)을 명확히 한다.

## 범위
- 대상 제품: LogJoiner (CloudWatch Logs 다단계 조인 분석 CLI)
- 범위: MVP 구현 및 안정화 (향후 로드맵 기능은 별도 에픽으로 분리)

## 기준 설정 파일 계약 (`config.yaml`)
- 본 백로그는 아래 키를 포함한 설정 파일을 읽어 실행하는 방식을 기준으로 한다.
- 기준 샘플 파일: `config.sample.yaml`
- 필수/권장 키
  - `log_group_name`: CloudWatch 로그 그룹 경로
  - `extract_pattern`: 1차 추출 패턴 (예: `ERROR`)
  - `output_file`: 기본(레거시/폴백) 최종 CSV 파일명
  - `analysis_steps[].name`: 스텝명
  - `analysis_steps[].query`: CloudWatch Logs Insights 쿼리 (멀티라인 지원)
  - `analysis_steps[].save_as`: 스텝 산출 CSV 파일명
  - `final_join_sqls[].name`: 조인 결과 식별자
  - `final_join_sqls[].query`: DuckDB SQL(문자열 또는 문자열 리스트)
  - `final_join_sqls[].save_as`: 조인 결과 CSV 저장 경로
- 쿼리 템플릿 규칙
  - `analysis_steps[].query` 내 `%s`는 실행 시 `extract_pattern` 값으로 치환한다.
  - 치환은 step 단위로 수행하며, 치환 실패 시 실행을 중단한다.
  - `final_join_sqls[].query`의 `FROM/JOIN` 대상은 `analysis_steps[].name`으로 참조 가능하다.

## Epic 1. CLI 입력 및 시간 변환 표준화
**목표**
- 사용자는 KST 기준으로 시간을 입력하고, 시스템은 UTC로 안전하게 변환해 조회한다.

### Story 1-1. CLI 기간 입력
- **사용자 스토리**: 분석가로서 `--start`, `--end`를 `yyyy-MM-dd HH:mm:ss` 형식으로 입력하고 싶다. 그래야 별도 계산 없이 조회를 시작할 수 있다.
- **수용 기준**
  - 기본 실행에서는 형식이 올바를 때만 실행된다.
  - `--skip-extract` 사용 시 `--start`, `--end`는 생략 가능하다.
  - 형식 오류 시 즉시 종료되고 사용 예시를 출력한다.

### Story 1-2. KST -> UTC 변환
- **사용자 스토리**: 운영자로서 KST 입력값이 CloudWatch API 호환 UTC 타임스탬프로 변환되길 원한다.
- **수용 기준**
  - 변환 로직이 표준 시간대 처리 라이브러리를 사용한다.
  - 변환 결과가 시작/종료 모두 정확히 적용된다.

### Story 1-3. 기간 유효성 검증
- **사용자 스토리**: 사용자로서 잘못된 기간(예: 시작 > 종료)을 사전에 막고 싶다.
- **수용 기준**
  - `start > end`면 실행되지 않는다.
  - 오류 메시지에 수정 가이드를 포함한다.

## Epic 2. YAML 기반 시나리오/설정 관리
**목표**
- 쿼리/스텝/출력 필드를 코드 변경 없이 YAML로 관리한다.

### Story 2-1. YAML 스키마 검증
- **사용자 스토리**: 분석가로서 필수 키 누락/오타를 실행 전에 검증하고 싶다.
- **수용 기준**
  - 필수 키(`log_group_name`, `extract_pattern`, `analysis_steps`) 누락 시 실패한다.
  - `final_join_sqls` 또는 `final_join_sql` 중 하나는 반드시 포함되어야 한다.
  - 오류 메시지에 누락 키 목록이 표시된다.

### Story 2-2. 멀티라인 쿼리 지원
- **사용자 스토리**: 분석가로서 YAML 멀티라인(`|`) 쿼리를 그대로 사용하고 싶다.
- **수용 기준**
  - 멀티라인 쿼리를 손실 없이 로딩한다.
  - CloudWatch 실행 시 줄바꿈/공백으로 인한 문법 오류가 발생하지 않는다.
  - `%s` 플레이스홀더가 있는 쿼리는 `extract_pattern`으로 치환되어 실행된다.

### Story 2-3. 스텝별 결과 파일명 제어
- **사용자 스토리**: 운영자로서 스텝별 중간 산출물 파일명을 설정에서 제어하고 싶다.
- **수용 기준**
  - 각 스텝 결과가 YAML 지정 파일명으로 저장된다.
  - 파일 충돌 시 덮어쓰기 정책 또는 고유 파일명 정책이 일관되게 동작한다.

### Story 2-4. 최종 출력 파일명 제어
- **사용자 스토리**: 사용자로서 최종 결과 파일명을 `output_file`로 지정하고 싶다.
- **수용 기준**
  - `final_join_sqls[].save_as` 미지정 레거시 형식에서 `output_file`이 폴백 경로로 동작한다.
  - 경로가 유효하지 않으면 명확한 오류 메시지를 제공한다.

## Epic 3. CloudWatch 다단계 추출 파이프라인
**목표**
- n단계 스텝을 순차 실행해 ID 기반 추적을 확장한다.

### Story 3-1. n단계 스텝 실행
- **사용자 스토리**: 분석가로서 2단계 이상 파이프라인을 구성하고 싶다.
- **수용 기준**
  - 스텝 순서대로 실행된다.
  - 이전 스텝 결과가 다음 스텝에 전달된다.

### Story 3-2. 비동기 쿼리 상태 처리
- **사용자 스토리**: 운영자로서 CloudWatch Insights 비동기 상태를 안전하게 처리하고 싶다.
- **수용 기준**
  - `Scheduled`, `Running`, `Complete`, `Failed`, `Cancelled`, `Timeout` 상태를 처리한다.
  - 실패 상태에서 원인 메시지를 출력한다.
  - 1회 조회 결과가 상한에 도달하면 시간 구간을 자동 분할해 재조회한다.

### Story 3-3. 재시도 및 실패 처리
- **사용자 스토리**: 운영자로서 일시 장애 시 자동 재시도를 적용하고 싶다.
- **수용 기준**
  - 재시도 횟수/대기 정책이 설정 가능하다.
  - 최대 재시도 후 실패 시 명확한 종료 코드와 메시지를 제공한다.

## Epic 4. 로컬 스테이징 및 DuckDB 조인
**목표**
- 중간 결과를 CSV로 스테이징하고 DuckDB SQL로 조인/정제한다.

### Story 4-1. 단계 결과 CSV 저장
- **사용자 스토리**: 데이터 엔지니어로서 스텝별 결과를 즉시 CSV로 저장하고 싶다.
- **수용 기준**
  - 모든 스텝 완료 후 CSV 파일이 생성된다.
  - 헤더/인코딩/컬럼 정합성이 유지된다.

### Story 4-2. DuckDB 최종 조인 SQL 실행
- **사용자 스토리**: 분석가로서 `final_join_sqls`를 로컬에서 순차 실행하고 싶다.
- **수용 기준**
  - `final_join_sqls` 각 항목은 `name/query/save_as`를 가진다.
  - 각 항목의 `query`는 순차 실행 가능하며 마지막 SQL 결과가 해당 항목의 결과가 된다.
  - `analysis_steps[].name`을 DuckDB 소스 테이블로 참조할 수 있다.
  - 조인 실패 시 SQL 오류 위치를 포함해 메시지를 출력한다.

### Story 4-3. 시간순 정렬 보장
- **사용자 스토리**: 사용자로서 최종 결과가 `@timestamp` 오름차순이길 원한다.
- **수용 기준**
  - 최종 결과 정렬이 항상 적용된다.
  - 정렬 키 누락 시 실행 전에 유효성 검증 에러를 반환한다.

## Epic 5. 최종 출력 및 운영 가시성
**목표**
- 사용자가 바로 공유 가능한 결과물과 실행 요약 정보를 제공한다.

### Story 5-1. 최종 SQL 결과 컬럼 사용
- **사용자 스토리**: 사용자로서 `final_join_sqls`의 마지막 결과 컬럼을 그대로 CSV로 받고 싶다.
- **수용 기준**
  - 별도 컬럼 필터링 없이 항목별 마지막 SQL 결과 스키마가 그대로 저장된다.
  - `save_as` 기준으로 복수 CSV가 생성된다.

### Story 5-2. UTF-8-BOM CSV 출력
- **사용자 스토리**: 사용자로서 엑셀에서 한글 깨짐 없이 파일을 열고 싶다.
- **수용 기준**
  - 최종 CSV가 `save_as`(또는 폴백 `output_file`) 이름으로 UTF-8-BOM 저장된다.
  - 샘플 한글 데이터로 수동 검증 가능하다.

### Story 5-3. 실행 요약 출력
- **사용자 스토리**: 운영자로서 실행 결과(행 수, 파일 경로, 소요 시간)를 확인하고 싶다.
- **수용 기준**
  - 종료 시 요약 로그를 출력한다.
  - 실패 시 실패 지점(스텝 번호/이유)을 함께 출력한다.
